## §3. 3축 비교 절차

### 3-0. 정비 목적

39편 분석 결과를 모듈·P4 지침에 반영. P4 집필자가 명확·빠르게·일관·누락 없이·압박 없이 집필 가능하도록.

### 3-1. 입력

메인 호출문에서 다음 명시:
- 점검 대상 (특정 §·특정 유형·특정 영역)
- 점검 질문 (4건 또는 단일)
- 기대 출력 형식

### 3-2. 3축 비교

```
[축 1] 39편 분석 자료 본문
   ↓
[축 2] 현 지침 표기
   ↓
[축 3] git 정비 이력 (의도된 차이 식별)
   ↓
[결과] 4종 출력
```

### 3-3. 차이 분류

| 분류 | 처리 |
| --- | --- |
| **의도된 차이** | git 정비 이력에 명시 → 현 지침 유지 |
| **신규 차이** | 분석 변동 + 정비 이력 미반영 → 정정 권장 |
| **분석 모호** | 분석 자료 불명 → 본문 점검 권장 |

### 3-4. 정비 원칙 (CLAUDE.md 정합)

- 줄이는 방향 우선
- 효과 없는 신설 회피 (기존 표현 정밀화 우선)
- 원인 단 해결 우선 (방지 지침 신설은 마지막 수단)
- 신설 불가피 시 기존 섹션에 한 줄 압축

### 3-5. 출력 형식

```markdown
## 정비 후보 — N건

### 후보 1: [영역명]
- **현 상태:** [현 지침 본문 + 위치]
- **분석 자료:** [39편 분석 결과 + 출처]
- **차이:** [정합·정정·신규 설계·폐기]
- **정비 이력:** [관련 git 커밋 또는 "없음"]
- **권장:** [본문 변경안 또는 메인 결정 요청]
```

## §4. 출력 4종

| 분류 | 처리 |
| --- | --- |
| **정합** | 현 지침 ↔ 분석 자료 일치 → 유지 |
| **정정** | 차이 발견 → 본문 정정 권장 |
| **신규 설계** | 분석 자료에 본질 영역 발견 → 신규 신설 (메인 7점검 + 유저 승인) |
| **폐기** | 자연 발생 영역 → 지침 폐기 권장 |
