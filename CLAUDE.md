## 핵심 금지 원칙

1. **지침 내부 용어를 대본에 노출하지 않는다** — SL, BR, ML, A극/B극, 피벗, 이문호공, 해소 비유, 역설엔진 등
2. **지침의 예시 문구를 대본에 그대로 복사하지 않는다** — 원리를 이해하고 새 표현을 생성
3. **확인되지 않은 수치·출처를 단정하지 않는다** — 웹 검색으로 팩트 확인 후 사용
4. **GATE를 건너뛰지 않는다** — 각 GATE 체크리스트 전 항목을 실제로 검증
5. **욕설 사용 금지** — "미쳤다", "대박" 수준의 구어 감탄만 허용

## 지침 수정 요청 시 처리 원칙

지침 본문(`지침/*.md`, `.claude/agents/*.md`, `.claude/commands/*.md`)에 대한 **모든 의미·구조 수정 요청**은 메인이 먼저 `대본분석` 에이전트(모드 1) 호출 여부를 유저에게 확인하고 진행한다.

**예외 (확인 없이 메인 직접 처리):** 명백한 표면 수정만

- 오타·구두점·공백·들여쓰기·마크다운 형식

**확인 시 메인 권장안 제시:**

메인은 수정 범위·복잡도·연쇄 영향을 검토하여 권장안을 함께 제시한다.

- 단순 신설·표현 보완·단일 섹션 수정 → 메인 직접 처리 권장
- 복수 섹션 동시 수정·시스템 구조 변경·연쇄 동기화 다수 발생 → 에이전트 호출 권장

**처리 흐름:**

1. 유저: 지침 수정 요청
2. 메인: 수정 의도 파악 후, 에이전트 호출 여부를 권장안과 함께 유저에게 확인
3. 유저: 메인 직접 처리 / 에이전트 호출 중 선택
4. 에이전트 호출 선택 시: 보고서 반환 → 합의 → 수정·이력 기록·커밋
5. 메인 직접 처리 선택 시: 변경안 보고 → 승인 → 수정·이력 기록·커밋

**유저 강제 지정 가능:**

- "대본분석 에이전트로 점검해라" → 강제 호출 (확인 생략)
- "에이전트 거치지 말고 바로 수정해라" → 메인 직접 처리 (확인 생략)

## 대본 파일 헤더 규칙

모든 대본 파일은 최상단에 아래 형식의 헤더를 반드시 포함한다.

```markdown
# 대본: [주제명]

> 제목: [최종 확정 제목]
> 썸네일: "[썸네일 메인 카피]"
> 서사 유형: [이미지 강화형 / 반전형 / 공감형 등]
> 목표 길이: [X~Y분]
```

- `목표 길이`는 **분(分)으로 표기**한다 (공백 포함 분당 약 300자 기준)
  - 예: `목표 길이: 12~13분`
- 헤더 다음에 `---` 구분선을 넣고 본문을 시작한다
- 제목·썸네일은 P1에서 유저가 선택한 확정안을 그대로 기재한다

## 대본 본문 섹션 헤더 표기 규칙 (2026-05-08 신설)

본문 헤더에 사용한 양식·전략을 압축 표기하여 검증 시 식별 가능하게 한다.

- `## 도입부` / `## 본문` (양식 표기 없음)
- `### SL{N} — 키워드 [구조 라벨]` (예: SL7 — 경제 인질화 (4차 방어망 · 정점 · Aha))
- `### 피벗` (모순형 한정)
- `### 본문 결산 ({양식})` (양식 = 유형별 4종 중 1종 — A·B·C·D)
- `## 결론부 ({결론 브릿지 어법 1~3종} / {엔딩 유형 1~2종})`
  - 예: `## 결론부 (결론 브릿지 ④ 회수 / 엔딩 ⑥ 통찰형 + ⑤ 위트반전 혼합)`

**금지:** `## 엔딩` (옛 표기) / `## 결론부 (엔딩)` (동의어 표기 — 결론부 ⊃ 엔딩 정의 위반)

## 대본 파일 저장 규칙

**폴더 생성 시점:** P1에서 주제가 확정되면(GATE 1 통과 시) `대본/NNN_주제명 (서사 유형)/` 폴더를 즉시 생성한다.

- **NNN은 3자리 번호** (001, 002, … 099, 100, …) — 100개 이상 확장을 전제한다
- **서사 유형 표기 (2026-05-12 신설):** 폴더명 끝에 괄호로 P1에서 확정된 서사 유형 명시 — 모순형 / 이미지 강화형 / 과정 추적형 중 1종. 예: `010_북한 핵 딜레마 (모순형)` / `002_소금 (이미지 강화형)` / `004_잉글랜드 vs 스코틀랜드 (과정 추적형)`. 폴더명만 봐도 유형 식별 가능하도록 한다

```text
대본/[NNN_주제명 (서사 유형)]/
  [퀵서치] 주제명.md                  ← P1 완료 시 저장
  [팩트리포트] 주제명.md              ← P2 완료 시 저장
  [조감도] 주제명.md                  ← P3 완료 시 저장
  [대본] 주제명.md                    ← P4 완료 시 저장 (최신본, 타임스탬프 없음)
  [낭독대본] 주제명.md               ← P5 완료 시 저장
  히스토리/
    [대본] 주제명_YYYYMMDD_HHMMSS.md  ← 대본 수정 전 이전 버전들 (타임스탬프 부여)
```

### Phase별 저장 파일

| Phase | 파일명 | 내용 |
|---|---|---|
| P1 | `[퀵서치] 주제명.md` | 핵심 팩트 리포트 + 제목/썸네일 후보 + 확정안 |
| P2 | `[팩트리포트] 주제명.md` | 딥리서치 팩트 리포트 (출처·마킹 포함) |
| P3 | `[조감도] 주제명.md` | 구조 보고 전문 |
| P4 | `[대본] 주제명.md` | 대본 본문 (루트 최신본 — 타임스탬프 없음) |
| P5 | `[낭독대본] 주제명.md` | 발음 변환 목록 + 대본 원문 |

**수정 시 반드시 지켜야 할 순서 (2026-05-08 갱신 — 파일명 형식 변경):**

1. 기존 대본 파일(`[대본] 주제명.md`)을 `히스토리/` 폴더로 이동하면서 **타임스탬프를 부여**한다 — `히스토리/[대본] 주제명_YYYYMMDD_HHMMSS.md` (덮어쓰기 금지)
2. 수정된 내용을 `대본/[NNN_주제명]/[대본] 주제명.md` 에 저장 (루트 최신본은 항상 동일 파일명, 타임스탬프 없음)
3. `대본/[NNN_주제명]/` 폴더 루트에는 최신본 1개(`[대본] 주제명.md`)만 존재해야 함
4. **새 파일 최상단에 수정 내역 주석을 반드시 삽입한다**

**수정 내역 주석 형식 (파일 맨 첫 줄):**

```markdown
<!-- [수정] YYYY-MM-DD HH:MM | 수정 내용 한 줄 요약 (예: 3장 팩트 보강, 오프닝 훅 교체, 전체 퇴고) -->
```

- 수정 항목이 여러 개면 쉼표로 구분하여 한 줄에 작성
- 이전 수정 주석은 히스토리 파일에 보존되므로 새 파일에는 최신 주석 1개만 유지

- 히스토리 폴더가 없으면 첫 수정 시 자동으로 생성
- 파일명 형식: 루트 = `[대본] 주제명.md` (최신본, 타임스탬프 없음) / 히스토리 = `[대본] 주제명_YYYYMMDD_HHMMSS.md` (구버전, 타임스탬프 부여)
- **기존 대본(001~008) 점진적 전환:** 기존 폴더는 그대로 유지하고, 다음 수정 시점에 새 규칙 적용. 신규 대본(010~)은 처음부터 새 규칙
